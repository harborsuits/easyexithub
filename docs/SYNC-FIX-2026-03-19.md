# Easy Exit Pipeline ↔ Calendar Sync Fix
**Date:** 2026-03-19 10:50 EDT  
**Author:** Atlas  
**Status:** ✅ FIXED

## Problem Summary

You observed that the **Pipeline board** showed many leads with follow-up metadata (voicemail, callback, etc.), but the **Calendar view** only showed one scheduled item.

### Root Cause

Two different data sources were not synced:

1. **Pipeline board** reads `leads.status` + communication metadata  
   - Shows conversation state: "voicemail", "callback requested", "contacted"
   - This is metadata **about the last call**, not a scheduled event

2. **Calendar view** reads `leads.next_followup_date`  
   - Only renders leads with an actual timestamp
   - Manual reschedules in the UI updated `leads.next_followup_date` only
   - **But the follow-up engine reads from `follow_ups.scheduled_for`**

### The Disconnect

```
Manual reschedule in UI → leads.next_followup_date updated
                                     ↓
                          Calendar shows the change
                                     ↓
                          But follow_ups table untouched
                                     ↓
                          Follow-up engine never sees the new date
```

Result: Manual reschedules were **cosmetic only**. The engine would still use the old `follow_ups.scheduled_for` timestamp (or no timestamp at all if the follow-up was never created).

## The Fix

### Changed File
**`~/Projects/easyexithub-main/src/components/pipeline/PipelineCalendar.tsx`**  
Lines 149-179 (reschedule mutation)

### What It Does Now

When you reschedule a lead in the calendar:

1. **Cancel old pending follow-ups** for that lead
   ```ts
   supabase.from('follow_ups')
     .update({ status: 'canceled', canceled_at: now, notes: 'Superseded by manual reschedule' })
     .eq('lead_id', leadId)
     .eq('status', 'pending');
   ```

2. **Create new follow-up row** with the new date
   ```ts
   supabase.from('follow_ups').insert({
     lead_id: leadId,
     kind: 'callback',
     source: 'manual_reschedule',
     status: 'pending',
     priority: 50,
     scheduled_for: new Date(date + 'T09:00:00-05:00').toISOString(), // 9 AM ET
   });
   ```

3. **Update calendar display field**
   ```ts
   supabase.from('leads').update({ next_followup_date: date }).eq('id', leadId);
   ```

Now the calendar, the engine, and the UI are **all reading from the same source of truth**.

## Next Steps (Optional Enhancements)

If you want **even more automation**, consider:

### 1. Auto-Populate Calendar for Every Lead with Metadata
Right now, only leads with a **real scheduled date** appear on the calendar.

You could add a default follow-up date for every lead that has `follow_up_needed = true` but no `next_followup_date`:

```sql
update leads
set next_followup_date = (last_contact_date::date + interval '4 days')::date
where follow_up_needed = true
  and next_followup_date is null
  and status != 'dead';
```

Then create matching `follow_ups` rows for the engine.

### 2. Guard Against Ghost Follow-Ups
The webhook can sometimes create a follow-up for a lead that's already dead (race condition when Vapi sends intermediate status before final "not-interested" outcome).

Add a guard in the webhook before creating follow-ups:
```ts
// Check lead status before creating follow-up
const { data: lead } = await ee.from('leads').select('status').eq('id', leadId).single();
if (lead.status === 'dead' || lead.status === 'dnc') {
  console.log(`[ee-webhook] Skipping follow-up for dead/dnc lead ${leadId}`);
  return null;
}
```

### 3. Sync Pipeline Temperature to Follow-Ups Table
If you want priority tiers on the calendar (hot = red border, warm = orange, etc.), you could add a `temperature` column to `follow_ups` and set it when creating the row.
